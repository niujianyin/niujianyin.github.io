2018世界杯前端总结

发表于 2018-07-25

2018足彩世界杯
1.PC项目概述:
项目分为首页，单场页和专题页，世界杯赛程页，世界杯赛事包（展示，购买，支付）， 订单中心，聚合了登录，比赛信息，小炮，球通，新闻，预测推荐，数据，欧洲赔率，亚洲盘口,大小盘口情报等模块
2.WAP项目概述:
项目分为首页，单场页和前瞻页，世界杯赛事包（展示，购买，支付）,聚合了登录，比赛信息，小炮，球通，新闻，数据等模块
充值
3.小炮APP（android, iOS）内嵌页项目概述:
项目分为首页，单场页和前瞻页，世界杯赛事包（展示，购买，支付）,聚合了比赛信息，小炮，球通，新闻，数据等模块
登录模块
4.新浪体育APP（android, iOS）内嵌页项目概述:
项目分为首页，单场页和前瞻页，聚合了比赛信息，小炮，球通，新闻，数据等模块

5.项目亮点-难点-问题-优化:
在前端层面，项目做到了代码规范，前后端分离，工程自动化，组件化，性能优化，构建了高效源码目录组织结构，实现可以在其它项目中复用的
12个基础组件、26个业务组件，解决线上代码缓存、体积优化、本地开发环境代理请求等需求，提升了开发效率，便于迭代和维护，更加结构化，提高项目的性能，nodejs自动化编译代码，部署项目。
架构:
MVVM框架：Vue.js （版本2.x）
状态管理：Vuex
前端路由：Vue Router
服务端通讯：axios、jsonp
移动端滚动库：better-scroll
构建工具：webpack 2.0/fis3
源码：es6
离线存储：localForage
图表：echarts
轮播图：swiper
模板引擎：art-template
css预处理 less/stylus

12个基础组件
1.confirm：确认对话框组件
2.alert: 确认消息组件
3.toast: 显示消息组件
4.listview：列表组件
5.loading：加载态组件
6.no-result：无结果展示组件
7.scroll：移动端滚动组件
8.slider：轮播图组件
9.login：登录组件
10.baseMixin 基础混合组件
11.z5Mixin 小炮购买混合组件
12.plMixin 赔率混合组件

2.block：占位组件
3.brand：导航链接组件
4.feed：新闻feed流组件
5.m-header：页面头部组件
7.match-list：小炮列表组件
8.tab：导航栏组件
9.landscape：横屏提示组件

11.match-info：单场比赛信息组件
12.plan-item：球通文章展示组件
13.hot-plan：球通热门推荐组件
14.ad：广告展示组件

难点:
项目结构设计，设计到糅合之前小炮多个页面到一个页面展示，组件的拆分、整合与复用，组件颗粒划分，页面和文件资源部署和缓存问题

问题:
1.项目需求变化频繁，导致有新增需求，运营维护需求都的开发来修改
2.APP端需求区别化，调试困难

优化:
1.前端开发流程规范明确：项目需求分析 – 项目目录结构设计 – 路由拆分 – 页面开发 – 组件开发 – 自动化部署
2.统一HTTP 库，屏蔽后端接口的差异化，拦截请求和响应，良好的RESTful API 设计
3.优化了前端开发体验，现在前端项目是使用Webpack构建的，构建速度慢，node是单线程的，通过缩小文件搜索范围，使用 DllPlugin（打包出一个个单独的动态链接库文件），HappyPack（分解任务和管理线程），ParallelUglifyPlugin（开启多个子进程压缩代码），优化了构建速度；使用自动刷新，开启模块热替换，优化使用体验
4.优化了前端代码输出质量：
在网络加载层面的优化，减少用户能感知到的加载时间，区分环境（开发和部署），压缩代码，使用 Tree Shaking（剔除 JavaScript 中用不上的死代码)，提取公共代码（多个页面公共的代码抽离成单独的文件），按需加载（分割代码的功能来实现按需加载）
在运行时的效率优化代码，使用 Prepack（是一个部分求值器，编译代码时提前将计算结果放到编译后的代码中，而不是在代码运行时才去求值。） 输出性能更高的 JavaScript 代码。
5.优化工作流程，项目开发前提供所需设计图，测试版本时间和上线时间，明确产出物，让前端工作可以按照预定的时间线执行
